# Code Wiki

> 基于 Karpathy 的 LLM Wiki 架构构建的个人/团队知识库系统

**[English](README.md)** | **[中文](README_CN.md)**

---

## 概述

Code Wiki 是一个用于构建个人/团队知识库的 Claude Code Skill。它基于 Karpathy 的 LLM Wiki 模式，在用户和原始文档之间创建结构化、累积的维基系统。

**核心理念**：知识不是每次查询时重新发现，而是每次摄取时累积构建。

### 核心特性

- 🗂️ **结构化存储**：纯文件系统，无外部依赖
- 🔄 **增量摄取**：每次源文档摄取触达 10-15 个维基页面
- 🔗 **知识图谱**：实体关系、概念关系、代码路径关联
- 🌐 **中文优化**：完整的中文命令和自然语言触发
- 🔍 **智能查询**：带引用存回功能的问答系统
- 🧹 **健康检查**：孤儿页面、链接完整性、代码路径验证
- 📁 **项目集成**：支持在项目目录创建维基，便于版本控制

## 快速开始

### 1. 初始化维基

```bash
# 默认位置 ~/.openwiki
/wiki init

# 在项目根目录创建（推荐开发项目使用）
/wiki init --path ./wiki

# 指定自定义目录
/wiki init --path ~/my-wiki
/wiki 初始化
```

### 2. 摄取文档

```bash
/wiki ingest ~/documents/article.md
/wiki 摄取 ~/documents/notes.md
```

每次摄取自动执行：
- 创建摘要页
- 提取/更新实体（人物、组织、项目）
- 提取/更新概念
- 更新实体/概念关系
- 检查是否需要创建综合分析或对比页
- 关联代码路径（如涉及）

### 3. 查询知识库

```bash
/wiki query 关于AI的最新发展
/wiki 查询 LLM和GPT的区别
/wiki 分析这个项目的架构
```

查询后，系统**会建议将答案存回维基**作为洞察。

### 4. 健康检查

```bash
/wiki lint
/wiki 检查
```

检查项：
- 孤儿页面
- 缺失链接
- Index 一致性
- 代码路径有效性
- 知识图谱完整性

### 5. 可视化知识图谱

```bash
/wiki visualize
```

生成交互式 HTML 图谱，支持拖拽、缩放。

## 目录结构

### 默认位置（用户主目录）

```
~/.openwiki/
├── raw/
│   ├── sources/           # 原始文档（不可变）
│   └── assets/            # 图片、资源
├── wiki/
│   ├── entities/          # 实体页
│   ├── concepts/          # 概念页
│   ├── summaries/         # 文档摘要
│   ├── synthesis/         # 综合分析
│   └── comparisons/       # 对比分析
├── schema/
│   └── CLAUDE.md          # 维基行为规范
├── index.md               # 内容索引
└── log.md                 # 操作日志
```

### 项目内创建（推荐开发项目）

```
your-project/
├── src/                   # 项目代码
├── wiki/                  # 维基目录（可提交到仓库）
├── .gitignore
└── README.md
```

项目内创建的好处：
- 可将知识库与项目代码一起版本控制
- 团队成员共享项目知识
- 支持 CI/CD 集成

## 页面格式

### YAML Frontmatter

```yaml
---
title: 页面标题
type: entity|concept|synthesis|comparison
tags: [标签1, 标签2]
created: 2024-01-01
updated: 2024-01-15
code_paths:
  - path: src/main.ts
    type: module
    description: 入口文件
related: [[entity-name]]
---
```

### 跨引用

```markdown
这是一个对 [[entities/john-doe]] 的引用
这也是对 [[concepts/llm]] 的引用
```

## 严厉拷问 FAQ（用户视角）

> 这部分用于回答“这个项目到底能不能用、值不值得用、出了问题怎么办”。

### 1) 现在能用吗？10 分钟能跑起来吗？

**回答**：当前版本可用于 Skill 级工作流演示（`init/ingest/query/lint/visualize`），但还不是“安装即生产”的应用发布态。  
**我们怎么解决**：
- 提供固定演示路径：`/wiki init --path ./wiki` → `/wiki ingest <file>` → `/wiki query <question>` → `/wiki lint`
- 补齐“最小可验证闭环”样例数据与预期输出（见下方行动计划）

### 2) 和直接让 LLM 写笔记相比，有什么不可替代价值？

**回答**：核心差异是“可累积结构化知识”而不是“单次回答”：
- 持久化页面结构（entities/concepts/summaries/synthesis/comparisons）
- 可追踪关系网络（实体关系、概念关系、代码路径）
- 查询结果可回写，知识随使用增长，而不是每次重建上下文

### 3) 质量怎么证明？有 benchmark 吗？

**回答**：目前没有公开 benchmark 数据，这确实是短板。  
**我们怎么解决**：
- 建立最小评测集（10~20 篇中英文样例）
- 定义核心指标：实体抽取准确率、关系有效率、查询引用覆盖率、lint 问题率
- 每次规则改动后做回归对比，防止质量倒退

### 4) 失败场景怎么处理？会不会把知识库写坏？

**回答**：当前依赖文件系统与日志，具备可审计性，但故障注入与回滚流程还需加强。  
**我们怎么解决**：
- 强化“写入前检查 + 写入后 lint + 失败日志”
- 对关键写入加备份策略（至少页面级快照）
- 增加“可恢复操作说明”（冲突、重复、格式错误）

### 5) Schema 变更后老数据怎么办？

**回答**：已有 `schema` 入口，但迁移规范不完整。  
**我们怎么解决**：
- 引入 schema version 字段与迁移文档
- 定义向后兼容策略（旧字段保留期、转换规则）
- 增加 migration checklist（升级前/后检查项）

### 6) 多人协作如何避免互相覆盖？

**回答**：项目内 `./wiki` 可纳入 git，但冲突治理流程仍需补齐。  
**我们怎么解决**：
- 约定目录级责任边界（如实体页命名与 ownership）
- 规范 PR 审查项：链接完整性、frontmatter 校验、冲突页人工确认
- 强制 `lint` 作为合并前检查

### 7) 隐私与安全怎么做？

**回答**：当前为本地文件系统优先，天然可控，但缺少显式的脱敏策略与数据边界声明。  
**我们怎么解决**：
- 增加敏感信息处理规范（PII、密钥、凭证）
- 在 ingest 前增加可选脱敏步骤
- 明确记录“哪些数据会被模型处理、如何保留日志”

### 8) 我为什么要信任自动生成关系？

**回答**：关系必须可追溯，否则不应视为高置信结论。  
**我们怎么解决**：
- 要求查询/分析优先提供来源引用
- 对低置信关系打标为“待确认”
- 通过 lint 检测“无来源支撑”的核心结论

### 9) 中文/多语言真的稳定吗？

**回答**：项目已提供中文命令触发与文档，但跨语言实体归一仍是优化重点。  
**我们怎么解决**：
- 增加中英混合语料测试集
- 建立术语映射与别名策略（中文名/英文名/缩写）
- 在 query 输出中展示同义实体映射

### 10) 半年后会不会维护崩掉？

**回答**：若没有指标和自动检查，确实容易退化。  
**我们怎么解决**：
- 将 `lint`、链接检查、页面完整性检查纳入周期流程
- 建立版本节奏（规则变更、schema 变更、兼容策略）
- 用文档+脚本固化“可重复维护流程”

## 30 天补短板行动计划（可执行）

### 第 1 周：可运行性
- 交付 `10 分钟上手闭环`（初始化→摄取→查询→检查）
- 增加示例输入与“预期输出示例”
- 完成标准：新用户按文档一次跑通

### 第 2 周：质量基线
- 建立最小评测集（中英文）
- 定义并记录核心指标与阈值
- 完成标准：能给出一次基线评测结果

### 第 3 周：可靠性与恢复
- 写入前后校验、失败日志、恢复指南
- 补充常见失败场景与处理手册
- 完成标准：至少 3 类故障可复现并恢复

### 第 4 周：协作与治理
- 完成 schema 版本与迁移规范
- 增加团队协作流程（PR 检查项 + lint 门禁）
- 完成标准：多人并行提交后可稳定合并
